## ytgrab hardening — backoff, throttle, optional cookies

- ask: the decided "harden to earn it" (option 2 of the "never rate-limited" card) is built. ytgrab no longer claims never. It now backs off for real when YouTube pushes back — a 429 or a "confirm you're not a bot" challenge makes it pause and lengthen the per-video gap for the rest of the run (doubling each hit, up to backoff_max=300s) instead of retrying into the same wall; it throttles the start-of-run channel enumeration (sleep_requests, default 2s between metadata requests, where it was a 1s burst); and it can carry your YouTube session via optional browser-cookie auth (cookies_from_browser / cookiefile, both off by default because turning them on trusts the tool with that session). The README says all of this honestly now. Accept to release the slice; speak if you have reservations and the words land here.
- try: run ytgrab as usual — `python3 ytgrab.py`, or the one-click launcher — against a channel. Downloads pace and resume exactly as the accepted core did; the start log line now ends "logged out". To exercise the new auth, set `"cookies_from_browser": "firefox"` (or your browser) in config.json and run: the start line ends "firefox cookies" and the run carries your logged-in session. The backoff only visibly fires if YouTube actually rate-limits — it logs `rate-limit signal — backing off…` and lengthens the gap; its escalation, de-bounce, and cap were exercised by a standalone test this session (commit 66a5952). The README's new "Rate limits — honest version" section is its own evidence the overclaim is gone.
